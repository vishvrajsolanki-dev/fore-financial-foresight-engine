## 1. Project Overview

FORE (Financial Foresight Engine) is an AI-powered personal finance product built to answer the one question everyday money apps avoid with honesty: *“Can I afford this — and what does it do to my future?”* Instead of stopping at a spending report, FORE keeps a live conversation going across three linked faces — **Past**, **Decide**, and **Ahead** — all reading and writing the same shared `financial_context` spine. The product’s differentiator is deliberate and demo-visible: when a user asks an affordability question, the LLM is not allowed to invent the math. It must call a real function, `canIAfford()`, which recalculates burn-rate and runway with the hypothetical purchase inserted, then narrates that returned number.

FORE was built for the **Cursor Community Hackathon Ahmedabad** (Saturday 18 July 2026, ~9:00–17:30, York IE APAC Pvt Ltd, Ambli, Ahmedabad) under the theme **“AI for Everyday Experiences.”** The event was the first official Cursor Community Hackathon in Ahmedabad: 100+ builders, an 8-hour day, teams of 3–4, mandatory Cursor usage, public GitHub, progressive commits, and a deployed demo before live on-stage pitches (about three minutes per team). Team name: **Artificial Alliance**. Members: **Vishvraj Solanki** (AI/ML/DS & database team lead), **Allen** (backend), **Drashti** (frontend), **Kavya** (QA tester). The team received roughly **$1,000 worth of sponsor/participant credits** (Cursor and partners such as ElevenLabs, Wispr Flow, Exa, Render) — participation credits, **not** a competition win. Status today: **completed hackathon ship with ongoing polish** (auth, CSV upload, marketing surfaces, demo storyboards/videos). Positioning: strong for recruiters and hiring managers evaluating full-stack + applied ML judgment; for fellow engineers interested in tool-calling UX; and for AI/ML practitioners who care about grounding LLMs in deterministic functions rather than vibes.

## 2. Tech Stack & Architecture

FORE runs as a **Next.js 14 App Router** TypeScript application (Tailwind, Recharts, Lucide) that hosts the three faces and API routes, plus an optional **Python FastAPI** ML microservice under `ml-service/`. Chat orchestration for Decide uses the **Groq API** with **Llama 3.1** and real tool-calling so the model must invoke `canIAfford` before stating affordability numbers. Without a Groq key, a deterministic fallback still calls the real math function — the team treated “honest numbers even when the LLM is offline” as a non-negotiable demo property.

There are two run modes. **Demo mode** (no `DATABASE_URL`) needs no auth: synthetic personas load client-side and the faces work with `npm run dev`. **Full-stack mode** adds JWT auth (short-lived access + revocable refresh cookies), **PostgreSQL via Prisma** (users, sessions, encrypted transactions), and real bank CSV upload with HDFC/ICICI/SBI-style column detection. Prisma CLI quirks matter operationally: Prisma reads `.env`, while Next reads `.env.local`, so schema pushes require an explicit `DATABASE_URL`.

ML can run **inline in TypeScript** inside Next (`ML_MODE=inline` when `RENDER_ML_BASE_URL` is empty) or against the Python service on Render. The architectural choice to put the Python service on an **always-on Render starter** (not Vercel Python serverless) was explicit: cold-start latency on the `canIAfford()` path would kill the live demo moment judges watch most closely. CORS on the ML service is scoped to Vercel/localhost, not `*`. Charts use Recharts (radar for archetype mix, running-balance line with projected zero marker). Shared types live in `types/financialContext.ts` as the contract between faces. Post-hackathon work also added marketing pages, onboarding (upload vs demo data), insights/reports/settings shells, PWA bits, and a storyboard-aligned Playwright trailer recorder.

**Tier-2 / not in main demo:** ElevenLabs (verdict narration) and Exa (price/context lookup) were designed as feature-flagged network calls and **were not part of the main shipped demo** — offered only as Tier-2 options.

Stack quick map: Next.js 14 + TypeScript + Tailwind + Recharts on Vercel; FastAPI ML on Render; Groq + Llama 3.1 tool-calling; Prisma + PostgreSQL for full-stack; JWT/bcrypt auth; synthetic persona JSON + benchmark cohort JSON checked into `data/`.

## 3. Features Built

**Past** decodes spend into an assigned archetype (five locked centroids via Euclidean distance — e.g. Disciplined Saver, The Foodie), burn-rate trend, runway / projected zero-balance date, radar visualization, and a running-balance chart with a projected ₹0 reference. Users enter via onboarding: upload a bank CSV or one-tap **Use demo data** (default Priya / Disciplined Saver persona among five synthetic ledgers of ~115–123 transactions each).

**Decide** is a plain-language affordability chat. Flagship question: *“Can I afford a ₹15,000 laptop next month?”* The UI shows suggestion chips, a live tool path, a **“✓ checked your numbers”** badge when the tool ran, and optional **Show the math** exposing the tool-trace. The function recalculates burn with the expense inserted and returns day-shift on the crisis/zero-balance date — the model narrates that result rather than guessing.

**Ahead** is a goal-pace calculator (target amount + date → on pace / behind) plus peer-benchmark percentile bars by category against a static modeled cohort. Critically, Ahead re-reads shared context after Decide writes a verdict, so a purchase decision visibly shifts goal pace — proving one spine, not three disconnected demos.

Around the core, the team shipped AppShell navigation (desktop sidebar + mobile bottom tabbar), auth flows (register/login/reset), encrypted persistence path for full-stack, CSV upload panel, Insights weekly brief and Reports surfaces, Settings (including appearance), marketing landing (*See your financial future clearly*), Features/Pricing/About, security docs, STARK planning set (team brief, contracts, 13 task handouts, roadmap), smoke/parity/unit verification scripts, and marketing demo storyboards plus desktop (~78s) and mobile (~57s) trailer recordings of the Past → Decide → Ahead spine.

## 4. Challenges & Solutions

The central product risk was **LLM hallucination on money math**. The solution was architectural, not prompt-only: force tool-calling to `canIAfford()`, show a verified badge, and keep a non-LLM fallback that still runs real math. A second risk was **cold-start latency** on the demo-critical path; the team locked Render always-on for Python rather than serverless Python on Vercel. A third challenge was **scope vs time** in a ~4.5–8 hour hackathon window: STARK planning cut multi-user persistence from the original demo spine, named honest ceilings (static benchmarks, linear burn projection, single-session goals), and deferred ElevenLabs/Exa to Tier-2 so the live minute stayed Past → Decide tool call → Ahead pace shift.

Integration friction included keeping all faces on one `financial_context`, Prisma env-file mismatch during schema push, TS↔Python duplication of classify/burn/canIAfford (parity/drift risk called out in later assessments), and demo-recording navigation bugs (hidden mobile tabbar links matching desktop selectors) fixed by targeting `.app-sidebar` vs `.mobile-tabbar`. Lessons: name the demo moment before building; put deterministic math behind the LLM; prefer boring always-on infra for the one path judges watch; cut Tier-2 ruthlessly; document contracts so parallel Cursor agents don’t collide.
