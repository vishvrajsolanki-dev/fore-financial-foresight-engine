## 1. Project Overview

FORE (Financial Foresight Engine) is an AI-powered personal finance product built to answer the one question everyday money apps avoid with honesty: *“Can I afford this — and what does it do to my future?”* Banking and budgeting apps usually stop at a report of what you already spent. FORE is designed as a continuing foresight loop: decode your past, decide on the next purchase with grounded numbers, then see how that choice moves your goals and peer position — all on one live data object.

The product is organized as **three linked faces** on a shared spine called `financial_context` (CONTRACT-001). No face is supposed to invent a number and display it without writing it back to that spine first. The deliberate differentiator — and the moment meant for judges and LinkedIn alike — is that when a user asks an affordability question, the LLM is **not allowed to invent the math**. It must call a real function, `canIAfford()`, which inserts the hypothetical expense into the burn-rate path, returns `affordable`, `day_shift`, and a new projected zero-balance date, then lets the model *narrate* that result. The UI makes the grounding visible with a **“✓ checked your numbers”** badge and an optional **Show the math** tool-trace.

**Product language in the app:** shell subtitle *Financial foresight engine*; landing hero *See your financial future clearly*; supporting line *Foresight for everyday money decisions — grounded in your real transactions*; close tagline *One question. Three faces. One real number behind every answer.* Brand treatment accents the **O** in FORE. Typography uses Manrope for headings and Lexend for body. Display currency can switch among **INR / USD / EUR** (display conversion only; underlying math stays INR).

FORE was built for the **Cursor Community Hackathon Ahmedabad** (Saturday 18 July 2026, ~9:00–17:30, York IE APAC Pvt Ltd, Ambli, Ahmedabad) under the theme **“AI for Everyday Experiences.”** It was the city’s first official Cursor Community Hackathon: 100+ builders, an 8-hour day, teams of 3–4, mandatory Cursor usage, public GitHub with progressive commits, and a deployed demo before ~3-minute live pitches. Team **Artificial Alliance**: **Vishvraj Solanki** (AI/ML/DS & database team lead), **Allen** (backend), **Drashti** (frontend), **Kavya** (QA tester). The team received roughly **$1,000 worth of sponsor/participant credits** (Cursor and partners such as ElevenLabs, Wispr Flow, Exa, Render) — participation credits, **not** a competition win.

**Who it’s for:** people who want a trustworthy “can I afford X?” answer before spending; India-first bank CSV users (HDFC / ICICI / SBI / Kotak-style exports) with optional demo personas for zero-friction trials. **Who cares about the build:** recruiters evaluating full-stack + applied ML judgment; engineers interested in tool-calling UX; AI/ML practitioners who care about grounding LLMs in deterministic functions. **Status:** completed hackathon ship with ongoing post-event polish (auth, encrypted persistence, marketing, insights/reports, storyboard demo videos).

## 2. Tech Stack & Architecture

FORE’s primary surface is a **Next.js 14 App Router** TypeScript app (Tailwind utility classes plus a design system in `globals.css`, Recharts, Lucide icons). It hosts marketing pages, auth, onboarding, the three faces under `app/(faces)/`, and API routes. An optional **Python FastAPI** microservice in `ml-service/` exposes `/classify`, `/burn-rate`, `/can-i-afford`, and `/health`. Decide chat is orchestrated primarily through **`POST /api/chat`** with a tool registry; a legacy `/api/decide` path also exists. Chat uses the **Groq API** with **Llama 3.1** and real tool-calling. If no LLM key is configured, a deterministic fallback still extracts an amount via regex and runs **`canIAffordMath`** so affordability numbers remain honest — a non-negotiable demo property for the team.

**Two run modes.** *Demo mode* (`DATABASE_URL` unset): middleware is a no-op, no login required, financial context lives in React state with browser storage helpers, and faces work with `npm run dev`. *Full-stack mode* (`DATABASE_URL` set): JWT access tokens (~15 min) plus revocable hashed refresh tokens (~7 days) in httpOnly cookies, bcrypt password hashing, PostgreSQL via Prisma (users, sessions, encrypted transaction descriptions), server-side session load for chat, and CSV upload via API where raw CSV is not persisted. Prisma CLI reads `.env` while Next reads `.env.local`, so schema pushes need an explicit `DATABASE_URL` in the shell.

**ML execution.** Default when `RENDER_ML_BASE_URL` is empty is **inline TypeScript** (`lib/ml/*` via `runInline`). With a Render base URL set, Next calls the Python service over HTTP. Render was chosen as an **always-on starter** specifically to avoid cold-start latency on `canIAfford()` — the path judges watch most. CORS is scoped to allowed origins, not `*`. The same three math surfaces exist in both TS and Python (classify / burn-rate / can-i-afford); later audits note parity drift risk, which is why parity scripts exist.

**Shared spine.** `types/financialContext.ts` defines session id, persona, monthly income, archetype (label + distances), burn_rate (daily average, trend slope, projected zero date, runway days, optional seasonal/band fields), transactions, goal (target, date, on_pace, pace_gap_days), `last_decide_verdict`, and benchmark rows. Charting uses Recharts (archetype radar; running-balance line with ₹0 and projected-zero markers). App shell: desktop sidebar (Past / Decide / Ahead + Insights / Reports + Settings) and mobile bottom tab bar (Past / Decide / Ahead / More). Feature flags gate self-verify, multi-currency, voice input/narration, Exa grounding, Ahead PNG export, etc.

**Tier-2 / not in main demo:** ElevenLabs verdict narration and Exa price/context lookup were designed as feature-flagged network calls and **were not part of the main shipped demo**.

**Stack quick map:** Next.js 14, TypeScript, Tailwind, Recharts, Groq + Llama 3.1, FastAPI/Python, Prisma, PostgreSQL, JWT/bcrypt, Vercel (web), Render (ML design target), Playwright for storyboard recording.

## 3. Features Built

### 3.1 Entry, marketing, and product shell

The marketing landing at `/` opens with the FORE brand, hero copy, and a product mock showing Past KPIs (e.g. Disciplined Saver + runway month) and a Decide affordability row. Deeper marketing pages cover Features, Pricing (Free / Pro ₹499 / Business ₹1499 plan cards), About, and a public Security docs page summarizing trust claims. Footer and nav link these surfaces for a product-shaped first impression, not just a raw hackathon demo URL.

Inside the app, **AppShell** keeps Past, Decide, and Ahead primary. A context bar shows the assigned archetype pill, currency selector (INR/USD/EUR), and auth controls. If the user has no loaded data and isn’t on Past or onboarding, the shell gates them with a prompt to add data. Onboarding at `/onboarding` presents equal choices: **Upload bank CSV** (monthly income + city tier Tier 1–3, drag/drop, HDFC · ICICI · SBI · Kotak copy) or **Use demo data** (loads Priya by default). Completion screen: “You’re ready” with links to Open Past or Ask Decide.

### 3.2 Past — spending decoded

Past’s job is to turn a ledger into a readable financial identity and runway story. Intro framing: *Your spending, decoded.* After classify + burn-rate run, the user sees:

- A **runway hero**: “If income stopped,” the projected zero-balance date, runway days, daily spend, and an honest disclaimer that the balance trend is a straight-line projection — not a claim of forecast accuracy.
- An **archetype card** assigned from spend mix (not user-picked): one of **Disciplined Saver**, **Impulsive Spender**, **The Foodie**, **Social Butterfly**, or **Balanced Spender**, with blurb + tip copy (e.g. Foodie → weekly food budget; Impulsive → 24-hour rule over ₹1,000). Pill: *Assigned from spend mix*.
- A **radar chart** of closeness to all five centroids (distance transformed for display).
- Stat tiles for **daily burn** and **balance trend** slope.
- A **running-balance line chart** with weekly sampling, a ₹0 reference line, and a projected-zero vertical marker.
- **CSV upload/replace** and a recent **transactions** table (date, merchant, amount, category); full-stack can recategorize and optionally learn merchant rules.
- Loading skeletons, empty, and error states for a finished-product feel.

ML behind Past: Euclidean nearest-neighbour classify against five locked centroids → `{ label, distances }`; burn-rate regressor → `{ daily_avg, trend_slope, projected_zero_balance_date, runway_days_if_income_stopped, … }`.

### 3.3 Decide — ask before you buy (core USP)

Decide is the flagship interaction. Intro: *Ask if you can afford something — I run the real numbers,* with explicit copy that every affordability answer comes from a live `canIAfford()` call. Empty state: *What are you deciding?* Suggestion chips include *Can I afford a ₹15,000 laptop next month?*, *Should I buy ₹5,000 headphones?*, and a simple *Hi*.

Flow: user asks → `/api/chat` with history (demo mode may send transactions + spine in the body; full-stack loads server session) → model may call tools from a registry that includes **`canIAfford`**, `getSpendByCategory`, `getBurnRate`, `getGoalStatus`, `setGoal`, `searchTransactions`, and `getBenchmark`. When `canIAfford` appears in the tool trace, the client applies a Decide verdict into shared context (`item`, `amount`, `day_shift`, `new_zero_balance_date`). UI shows **✓ checked your numbers** (and optional self-verify “(review)”), **Show the math** JSON, a Context drawer (balance, daily burn, runway, archetype), optional browser mic input (`en-IN`), a “Still checking…” slow state (~2s), and a request timeout (~15s). Rate limiting applies on the chat API. Fallback path without an LLM still runs real affordability math.

`canIAfford` return shape: `{ affordable, day_shift, new_zero_balance_date, explanation }` after inserting a hypothetical shopping outflow and re-running burn-rate.

### 3.4 Ahead — goals, peers, and Decide impact

Ahead answers *Where your money is going* using the same live context Decide just updated. Users set a **savings goal** (target amount + future date; defaults exist in the form) and hit **Check pace**. Pace math compares implied daily surplus (`monthly_income/30 − daily_avg`) to required savings per day; outcomes include on pace / behind pace / not reachable, with insight headlines and stats (days gap, need/day, surplus/day, burn). A **Recent DECIDE verdict** banner can show how the last purchase shifts zero-balance date and how many days further the goal moves when a verdict is present. **Peer benchmark** panel shows modeled percentiles by category (food, shopping, bills, entertainment, savings) for the user’s income bracket × city tier against authored tables in `data/benchmark.json` (5 brackets × 3 tiers), with explicit “modeled / not a live survey” disclaimer. Optional PNG export of the Ahead summary when the feature flag is on.

### 3.6 Secondary product surfaces

**Insights** builds a weekly foresight brief: spend/inflow overview, archetype, burn/runway, latest Decide, goal, peer snapshot, with deep links back to faces (full-stack can hit `/api/insights/weekly`). **Reports** offer date-range preview metrics (spend by category/merchant, cashflow, transactions) and CSV export (`fore-report.csv`). **Settings** cover profile, password/sessions, notification toggles, local plan selection, Light/Evening appearance, help links, account export, and delete account in full-stack mode. **Auth** includes email/password register/login, forgot/reset password, email verify, logout/refresh/me, and optional Google/Microsoft OAuth when env is configured.

**PWA:** web manifest names FORE, `start_url` `/past`, standalone display; service worker caches shell only (not HTML/`_next`/API); registration is production-oriented.

**Security/privacy (documented):** bcrypt; AES-GCM on sensitive description fields; raw CSV not stored; JWT payload kept minimal; user-scoped queries; LLM typically receives spine summaries rather than encrypted narrations; Groq key server-side; HTTPS in production; product disclaimer that FORE is not licensed financial advice. Full writeup in `docs/SECURITY.md`.

## 4. Challenges & Solutions

The central product risk was **LLM hallucination on money math**. The solution was architectural, not prompt-only: force tool-calling to `canIAfford()`, surface a verified badge and math trace, and keep a non-LLM fallback that still runs real math. A second risk was **cold-start latency** on the demo-critical path; the team locked Render always-on for Python rather than serverless Python on Vercel. A third challenge was **scope vs time** in a one-day hackathon: STARK planning deferred multi-user persistence from the original ultra-thin demo spine (later reintroduced as full-stack mode), named honest ceilings (static benchmarks, linear burn projection, single-session goals in the early cut), and kept ElevenLabs/Exa in Tier-2 so the live minute stayed Past → Decide tool call → Ahead pace shift.

Integration friction included keeping all faces honest to one `financial_context`, Prisma’s `.env` vs Next’s `.env.local` mismatch during schema push, TS↔Python duplication of classify/burn/canIAfford (parity/drift called out in assessments), chat rate limits and timeouts under demo pressure, and demo-recording navigation bugs where hidden mobile tab links matched desktop selectors — fixed by targeting `.app-sidebar` vs `.mobile-tabbar`. Lessons: name the demo moment before building; put deterministic math behind the LLM; prefer boring always-on infra for the one path judges watch; cut Tier-2 ruthlessly; document contracts so parallel Cursor agents don’t collide.
